Theme 03

多模型编排成主路：planner 加 worker，token 成本 15 倍下降 / Multi-model orchestration becomes the default

Box CEO Aaron Levie 强调 frontier model 当 planner，廉价模型当 worker，可以拿下 15x token 成本下降；Vercel CEO Guillermo Rauch 则从另一个角度：AI 把一切都还原成 code。

AL
Aaron Levie
Box CEO
@levie
中文

Levie 引 Cursor 的最新研究：让 frontier 模型当 planner 和 orchestrator，再配一个便宜的 workhorse 模型，长任务的总 token 成本能显著下降——具体到这篇报告是 15 倍。一项大任务真正需要 frontier 智能的只有少数几个节点：最初的拆解、关键设计决策和取舍；一旦被 planner 翻译成清晰的指令，剩下的部分交给便宜模型照做就行。

让 frontier 模型做 planner 和 orchestrator，再配一个更便宜的 workhorse 模型，整个项目的 token 成本可以显著下降，提升达 15 倍。大型任务里真正需要 frontier 智能的只有少数几个时刻：最初的拆解、关键设计决策以及某些取舍。

English

Levie cites new research from Cursor: a frontier model as planner, paired with a cheaper workhorse model, materially lowers token cost on long-horizon tasks. Only a few moments in a big task truly need frontier reasoning — the original decomposition, design decisions and key trade-offs — once those are collapsed into explicit instructions, cheaper models just have to follow them, yielding a 15× cost improvement.

Multi-model agentic systems clearly are the future. Great post by cursor. Their new research shows that a frontier model as a planner and orchestrator with a workhorse cheaper model can materially lower the cost of total tokens on a project, resulting in a 15X cost improvement.

Swyx 从 Alex Zhang 与 Omar Khattab 的 RLM 论文里挖出一段被埋掉的对比：frontier 模型训练其实有一个公开秘密——即使没有直接训练在 test 上，但只要在 test 的「近似集」上训练，就能在几乎任何 benchmark 上刷出想要的数字。一旦权重开源，但训练数据集和 RL environment 不开源，下游几乎没法判断模型是否专门针对某个 benchmark 做了优化，这就是「合理的可否认性」。

在 @a1zhang 与 @lateinteraction 的 RLM 论文里，藏着一段关于模型训练轨迹的对比。"frontier" 模型训练的一个公开秘密：即便没有直接训练在 test 上，你也能通过训练在 test 的近似分布上作弊，刷出想要的 benchmark 数字。但模型以开放权重发布时，99% 的情况下你拿不到能证明它是否针对某 benchmark 训练的数据集和 RL 环境，于是「合理的可否认性」就成立了。

English

very notable trajectory comparison writeup here buried in the RLM paper from @a1zhang and @lateinteraction. an open secret of "frontier" model training is that even without training on test, you can basically cheat by training on test lookalikes, enabling you to goalseek almost any benchmark number you want. however when they are released open weights, 99% of the time the norm is that you do not get the datasets/rlenvs that would easily show you if someone was training on Temu Tbench, so there is plausible deniability.

Theme 07

Agent 实战：评审 / 翻车 / 声音 / Agent in production: rubric, bugs and voice

Peter Yang 引用 trq212 关于 video short 的 self-preferential bias；trq212 自己承认 Discord bug 刚下线；Amjad Masad 则把「被封号是一种选择」反复用到差不多成了一句 slogan。

PY
Peter Yang
产品教练 / 创作者经济研究者
@petergyang
中文

Peter Yang 转述 Thariq 的实战经验：让一个 agent 生成视频短片，再让另一个 agent 对照 rubric 评审。短片「好不好」本来就没有标准答案，关键是让评审 agent 独立判断，因为模型天生对自己的输出更宽松——他们把这叫 self-preferential bias。

用一个 agent 完成工作，再用一个 agent 按 rubric 评审。短片「好不好」没有标准答案。你应该让一个独立的评审 agent 读 rubric、看片、再给反馈。我们把这种倾向叫做 self-preferential bias：模型对自己的输出更宽容，评审时自然会更宽松。

English

Peter Yang quotes Thariq on running video shorts: one agent should generate, a second should grade the output against a rubric — because the model that wrote the piece tends to be lenient judging itself, a self-preferential bias Thariq calls out by name.

Use one agent to do the work and another to review it against a rubric. For something like, 'Is this a good video short or not?' you don't have a deterministic answer. You should have a separate verification agent read a rubric, review the short, and give feedback. We call it self-preferential bias.
