中文

OpenAI 的 Codex 负责人 Thibault Sottiaux 甩出来一张一句话图表:Codex 的小毛病(papercuts)在往下掉,渗透率(也就是真实采用率)跟着往上走。memory 变好、prompt 变短、每个 token 能干的事变多——一句话把「为啥越用越离不开」讲清楚了。

English

OpenAI's Thibault Sottiaux (head of Codex) shares a one-line chart: as Codex papercuts (the day-to-day rough edges) drop, adoption rises. Better memory, shorter prompts, more utility per token — the model gets cheaper to use, and usage grows with it.

He frames it as the single most important non-benchmark signal for an agent: if you stop losing time to the harness, you start letting it touch more of your workflow.

Codex papercuts 📉 Codex adoption 📈

Better memory = Shorter prompts = More utility per token

Swyx 这条推讲的是:比「永远先 plan」更稳的默认动作,是「永远把任务框成问题」。这样模型就被邀请去反驳、去评估你的想法,或者直接给替代方案,而不是照字面执行那个其实不是你真正想要的指令。

实操上,在 prompt 末尾加一个问号,通常就够把这件事做对。

比起「永远先 plan mode」,更聪明的做法是把任务框成问题,让模型去 push back、给你评分或给替代方案,而不是盲目照你说的做(你说的往往不是你想的)。

English

Swyx argues the more useful default isn't "always use plan mode" — it's "always frame your task as a question". The model is then invited to push back, rate the idea, or suggest an alternative, instead of dutifully executing the literal ask that often isn't what you meant.

Concretely, appending a single "?" to the end of a prompt often does most of the work.

a smarter alternative to "always use plan mode": always frame your task as a question, so that the model is invited to push back and rate the quality of the idea/suggest alternatives, rather than blindly execute what you SAID to do (which is often not precisely what you MEANT)

Agent infra 与额度扩容 / Agent Infra, Skills API, and Doubled Limits

Vercel 把 agent filesystem 跟 sandbox 解耦 + Skills API 走 npm 路线,Replit 拿下 Shopify 渠道,Claude Cowork 额度翻倍——算力、存储、电商管道、预算四条线同时松绑

GR

Vercel CEO
@rauchg
中文

Vercel 这次放出来的能力:agent 的文件系统状态,可以独立于 sandbox 生命周期被读写和挂载。这是一套新建的虚拟存储层,跟 Builds、Functions、Sandboxes 这些算力产品解耦,但又可以随时挂回去。

English

Vercel: agent filesystem state can now be read, written, and mounted independently of the Sandbox lifecycle. They've built a virtual storage layer that's decoupled from — but attachable to — Builds, Functions, Sandboxes, and other compute products.

The implication: an agent's working memory survives across sandbox restarts, so you can ship stateful long-running jobs without re-plumbing the data plane each time.

Agent filesystem state can now be read, written and mounted independently of Sandbox lifecycle. We've been developing a novel virtual storage infrastructure solution for all our compute products. Storage that's decoupled, but attachable to Builds, Functions, Sandboxes & more.

企业 AI 团队最常见的错误是按今天的模型和价格去设计。思路:看 6 个月后——用脚手架补今天的弱点,赌下一代模型会接管脚手架。Build for the slope,不是 build for the snapshot。

English

Madhu Guru's framework for enterprise AI teams: don't build for today's model capabilities and price points. Think 6 months out. The trick is to scaffold around today's model weaknesses to push the frontier, while betting that the next generation of models will natively solve the scaffold — and then push the frontier again.

His catchphrase: build for the slope.

One of the most common mistakes I see enterprise AI teams make is building for today's model capabilities and price points. Think 6 months out. Models will be way smarter and cheaper. Scaffold around today's model weaknesses to push the frontier. Bet that the next generation of models will natively solve for the scaffold. Build for the slope.
