Theme 01

Managed Agents 把 harness 做成产品层 / Managed Agents Turns the Harness Into a Product

Anthropic 同日连发三篇:Managed Agents 把 agent 拆成 session/harness/sandbox 三个稳定接口、Claude Code 三月到 4-23 的 quality postmortem 自承 harness 累加改动的混乱、Claude Blog 把 self-hosted sandboxes + MCP tunnels 当 product 摆出来。同一天 Rauchg 把 HarnessAgent 发到 @aisdk 上——harness 这一层从 narrative 升级成产品表层,各家都在抢抽象。

A
Anthropic
Anthropic 官方博客
中文

Anthropic Engineering 把 Managed Agents 的架构帖摆出来。核心动作:把 agent 虚拟化成三个稳定的接口——session(append-only 日志)、harness(调用 Claude 并路由工具的 loop)、sandbox(代码执行环境)。底下实现可以换,接口不动。read() 系统调用不在乎盘是 1970 年的 disk pack 还是现代 SSD——同一个 trick,套到长程 agent 上。

帖子里最具体的自我承认:一段针对 Sonnet 4.5 写的 harness(加 context reset 来对抗「context anxiety」)到 Opus 4.5 变成死重——模型自己不再提前收摊了。Anthropic 的论点是结构性的,不光是产品级的:接口保持稳定,底下实现不断换。Managed Agents 是这个论点的 hosted 版本。

第二个动作:不再把容器当宠物养。第一版 Managed Agents 把 session/harness/sandbox 塞进同一个容器,容器挂掉 session 就没了,调试还得在装着用户数据的容器里开 shell。Anthropic 把它们拆开,harness 跑在 Anthropic 自己的基础设施上,sandbox 跑在客户网络里——工程师能真正调试,而不必碰用户数据。

Anthropic Engineering:Managed Agents 把 agent 拆成 session/harness/sandbox 三个虚拟层,接口稳定、底下可换;Sonnet 4.5 的 context reset 在 Opus 4.5 上变成死重,证明 harness 必须持续演化。

English

Anthropic Engineering drops the Managed Agents architecture post. The core move: virtualize an agent into three stable interfaces — session (append-only log), harness (the loop that calls Claude and routes tools), sandbox (where code runs). Implementations underneath can be swapped without disturbing the other two. Read() didn't care whether the disk was a 1970s pack or an SSD — the same trick, applied to long-horizon agents.

The most concrete admission in the post: a harness that worked for Sonnet 4.5 (context resets to fight 'context anxiety') became dead weight on Opus 4.5 because the model stopped wrapping up prematurely. Anthropic's argument is structural, not just product: keep the interfaces stable, keep swapping the implementations underneath. Managed Agents is the hosted service form of that argument.

A second move: stop treating the container as a pet. The first version of Managed Agents bundled session/harness/sandbox into one container, which meant failures lost the session and debugging required opening a shell inside user-data-bearing infra. Anthropic split them apart so the harness can run on Anthropic infra while the sandbox runs on the customer's network — and an engineer can actually debug without touching user data.

Get started with Claude Managed Agents by following our docs. A running topic on the Engineering Blog is how to build effective agents and design harnesses for long-running work. A common thread across this work is that harnesses encode assumptions about what Claude can't do on its own. However, those assumptions need to be frequently questioned because they can go stale as models improve.

As just one example, in prior work we found that Claude Sonnet 4.5 would wrap up tasks prematurely as it sensed its context limit approaching—a behavior sometimes called 'context anxiety.' We addressed this by adding context resets to the harness. But when we used the same harness on Claude Opus 4.5, we found that the behavior was gone. The resets had become dead weight.

We expect harnesses to continue evolving. So we built Managed Agents: a hosted service in the Claude Platform that runs long-horizon agents on your behalf through a small set of interfaces meant to outlast any particular implementation.

English

Same day, Claude Blog ships the product update on top of the architecture post: self-hosted sandboxes (public beta) and MCP tunnels (research preview). The pitch in plain terms — the agent loop, context management, and error recovery stay on Anthropic; tool execution moves into your infrastructure, or to one of the managed providers (Cloudflare, Daytona, Modal, Vercel). Files and repos don't leave your perimeter; credentials are injected at the network boundary so they never enter the sandbox.

Why this matters: every prior enterprise objection to long-running agents was a network boundary problem. Now you get Cloudflare microVMs with zero-trust secrets injection, Daytona long-running stateful computers, Modal's GPU-on-demand foundation, or Vercel's millisecond-startup sandbox with VPC peering. The agent loop runs on Anthropic, the substrate runs where your data already lives. Customer callouts in the post — Amplitude (Design Agent on Cloudflare), Clay (Sculptor on Daytona), Rogo (analyst agent on Vercel) — show the pattern is already in production at non-trivial customers.

Starting today, Claude Managed Agents can operate in a sandbox you control and connect to your private Model Context Protocol (MCP) servers. Both the sandbox where an agent executes tools and the services it reaches run within the established boundaries of your enterprise, under your security and runtime controls.

On the Claude Platform, self-hosted sandboxes is available in public beta and MCP tunnels in research preview (request access).

Same day's third Anthropic post — a public postmortem on Claude Code quality degradation from late March through April 23. Three independent changes, each on a different schedule, hit different traffic slices: (1) March 4 default reasoning effort dropped from high to medium to fight UI-feel-frozen latency; reverted April 7 after users said they'd rather opt into lower effort for simple tasks. (2) March 26 idle-session thinking-clear logic had a bug that re-ran every turn; fixed April 10. (3) April 16 system-prompt verbosity reduction, combined with other prompt changes, hurt coding quality; reverted April 20.

The honest read in the post: 'the aggregate effect looked like broad, inconsistent degradation' and the internal usage/evals didn't initially reproduce what users reported. Anthropic is resetting usage limits for all subscribers as of April 23. This is the harness-stability argument playing out inside the company's own product — when three small defaults shift on different days, the user-visible behavior is incoherent, and that's a harness-engineering failure, not a model failure.

Over the past month, we've been looking into reports that Claude's responses have worsened for some users. We've traced these reports to three separate changes that affected Claude Code, the Claude Agent SDK, and Claude Cowork. The API was not impacted. All three issues have now been resolved as of April 20 (v2.1.116).

Because each change affected a different slice of traffic on a different schedule, the aggregate effect looked like broad, inconsistent degradation. While we began investigating reports in early March, they were challenging to distinguish from normal variation in user feedback at first, and neither our internal usage nor evals initially reproduced the issues identified.

This isn't the experience users should expect from Claude Code. As of April 23, we're resetting usage limits for all subscribers.
