English

Anthropic Engineering posts the official postmortem for the April Claude Code quality complaints. The conclusion: API was unaffected, but three independent harness changes between March 4 and April 16 cumulatively degraded Claude Code, Claude Agent SDK, and Claude Cowork — and the way they stacked on different traffic slices made the aggregate look like generic, inconsistent quality drift.

The three changes: March 4 dropped Claude Code's default reasoning effort from high to medium (reverted April 7); March 26 introduced an idle-session thinking-clear bug that fired every turn (fixed April 10); April 16 added a verbosity-reducing system prompt instruction that hurt coding quality (reverted April 20). All three impacted Sonnet 4.6 / Opus 4.6, and the last one also impacted Opus 4.7.

The post is significant for what it admits without flinching: harness changes, not model regressions, were the cause. Internal usage data and evals did NOT initially reproduce the user-reported issues. As of April 23 they reset all subscriber usage limits. The takeaway for builders: when a model's behavior changes, the model is the variable you suspect first — but a week of harness changes can produce the same shape of complaint at fleet scale.

Over the past month, we've been looking into reports that Claude's responses have worsened for some users. We've traced these reports to three separate changes that affected Claude Code, the Claude Agent SDK, and Claude Cowork. The API was not impacted.

On March 4, we changed Claude Code's default reasoning effort from high to medium to reduce the very long latency—enough to make the UI appear frozen—some users were seeing in high mode. This was the wrong tradeoff. We reverted this change on April 7.

On March 26, we shipped a change to clear Claude's older thinking from sessions that had been idle for over an hour, to reduce latency when users resumed those sessions. A bug caused this to keep happening every turn for the rest of the session instead of just once.

On April 16, we added a system prompt instruction to reduce verbosity. In combination with other prompt changes, it hurt coding quality and was reverted on April 20.

As of April 23, we're resetting usage limits for all subscribers.

Anthropic Engineering 紧接着发 Managed Agents 架构帖,把 agent 拆成 session / harness / sandbox 三个可换的虚拟化组件;Claude Blog 在上面叠 self-hosted sandboxes + MCP tunnels;Boris Cherny 把 subagents 默认 background 化——同一套思路在 blog / service / CLI 三个 surface 一起铺。

A
Anthropic
Anthropic 官方博客
中文

Anthropic Engineering 在 quality postmortem 之后发了架构层面的 follow-up:Managed Agents,把 agent 拆成三个可换的组件——session(append-only 的事件日志)、harness(调 Claude + route tool call 的循环)、sandbox(执行环境)。任何一层的实现变了,不会扰动另外两层。

思路是从 OS 借的:设计约束是「为还没被想出来的程序」设计,正确做法是把硬件虚拟化成 process / file 这种通用到「程序不存在也能用」的抽象。Managed Agents 套同一套模式——对接口的形状有 opinion,对接口后面跑什么保持中立。

worked example 是 Claude Sonnet 4.5 的「context anxiety」(context 快满时主动收尾)。harness 层加 context reset 解决了。同一套 harness 跑到 Opus 4.5 上,这个行为消失了——那些 reset 成了 dead weight。架构接受这件事会一直发生:模型变强,今天的 harness 假设就过期;唯一耐用的只有接口本身。

'Don't adopt a pet' 是操作守则:不要对某一套 harness 实现产生感情,底下的模型一换,你的 pet 就不是 load-bearing 了。Managed Agents 是这个立场的服务化版本。

Anthropic Engineering 把 agent 拆成 session / harness / sandbox 三个虚拟化组件,每层可独立替换,不动其他层——OS 时代「为还没想出来的程序设计」思路的服务化版。

English

Anthropic Engineering follows up the quality postmortem with the architectural follow-up: Managed Agents, a hosted service that virtualizes an agent into three swappable components — a session (the append-only log of everything that happened), a harness (the loop that calls Claude and routes tool calls), and a sandbox (the execution environment). The implementation of any one can change without disturbing the others.

The framing is borrowed from operating systems: 'programs as yet unthought of' is the design constraint, and the right move is to virtualize hardware into abstractions (process, file) general enough for programs that don't exist yet. Managed Agents applies the same pattern to agents — opinionated about the shape of the interfaces, agnostic about what runs behind them.

The worked example is 'context anxiety' in Claude Sonnet 4.5 (wrapping up tasks as context fills). The fix was harness-level context resets. When the same harness ran on Claude Opus 4.5, the behavior was gone — the resets were dead weight. The architecture accepts that this will keep happening: as models improve, today's harness assumptions go stale, and the only durable thing is the interface.

The 'don't adopt a pet' line is the operational maxim: don't grow attached to a particular harness implementation, because the model underneath it will change and your pet will stop being load-bearing. Managed Agents is the explicit service-wrapped version of that stance.

A running topic on the Engineering Blog is how to build effective agents and design harnesses for long-running work. A common thread across this work is that harnesses encode assumptions about what Claude can't do on its own. However, those assumptions need to be frequently questioned because they can go stale as models improve.

As just one example, in prior work we found that Claude Sonnet 4.5 would wrap up tasks prematurely as it sensed its context limit approaching—a behavior sometimes called 'context anxiety.' We addressed this by adding context resets to the harness. But when we used the same harness on Claude Opus 4.5, we found that the behavior was gone. The resets had become dead weight.

Building Managed Agents meant solving an old problem in computing: how to design a system for 'programs as yet unthought of.' Decades ago, operating systems solved this problem by virtualizing hardware into abstractions—process, file—general enough for programs that didn't exist yet. The abstractions outlasted the hardware.

Managed Agents follow the same pattern. We virtualized the components of an agent: a session (the append-only log of everything that happened), a harness (the loop that calls Claude and routes Claude's tool calls to the relevant infrastructure), and a sandbox (an execution environment where Claude can run code and edit files). This allows the implementation of each to be swapped without disturbing the others.

Don't adopt a pet.
